## 1. Package Specification Review

**File:** `fault_correction_types.ads`

### Issue 1.1 — Missing Default for `Command_Response` Field

- **Location:** `fault_correction_types.ads`, line 20 (`Command_Response : Command.T;`)
- **Original Code:**
  ```ada
  Command_Response : Command.T;
  ```
- **Explanation:** Every other field of `Fault_Response_Config` has an explicit default initializer. `Command_Response` does not. In safety-critical flight code, uninitialized record fields risk using whatever value occupies memory (e.g., a stale command from a prior allocation). If `Command.T` has a valid default via Ada's default initialization, this is benign, but relying on implicit defaults for a *command to be issued during a fault response* is a design-level risk — an accidental default could fire a real command. An explicit default (e.g., `Command.T'(others => <>)` or a named "no-op" constant) would make intent clear and prevent accidental dispatch.
- **Suggested Code:**
  ```ada
  Command_Response : Command.T := (others => <>);
  -- or, preferably, a named constant:
  -- Command_Response : Command.T := Command.Empty;
  ```
- **Severity:** **High** — A default-initialized command struct sent to a dispatcher could trigger unintended behaviour on a spacecraft.

### Issue 1.2 — `Two_Bit_Padding_Type` Has No Use Guard or Documentation on Permitted Values

- **Location:** `fault_correction_types.ads`, line 26 (`type Two_Bit_Padding_Type is mod 2**2;`)
- **Original Code:**
  ```ada
  type Two_Bit_Padding_Type is mod 2**2;
  ```
- **Explanation:** This modular type allows values 0–3, but its only purpose is padding. There is no constraint or documentation ensuring padding is always zero when serialized/deserialized. In telemetry, non-zero padding bits can confuse ground tools or mask bit-rot. Consider adding a subtype or a comment specifying the expected value.
- **Suggested Code:**
  ```ada
  -- Padding type for status packet alignment. Padding fields must always be 0.
  type Two_Bit_Padding_Type is mod 2**2;
  subtype Two_Bit_Padding is Two_Bit_Padding_Type range 0 .. 0;
  ```
- **Severity:** **Low** — Informational/defensive. The template already sets format `U2` which handles serialization, but zeroing intent should be documented.

### 3.2 Record — `packed_fault_response_statuses.record.yaml`

#### Issue 3.2.1 — Hardcoded Four-Entry Status Record Limits Reusability

- **Location:** `packed_fault_response_statuses.record.yaml`, entire file
- **Original Code:**
  ```yaml
  description: Status information for 4 fault response entries.
  fields:
    - name: Status_0
      ...
    - name: Status_3
      ...
  ```
- **Explanation:** This record packs exactly 4 status entries (4 × 2 bits = 8 bits = 1 byte). The autocoded template in `gen/templates/fault_responses/name_status_record.record.yaml` already generates a dynamic-length status record with proper padding. This hardcoded 4-entry record appears to be a convenience type (e.g., for a packed telemetry sub-word). However, if the number of fault responses is not a multiple of 4, this type cannot represent the final partial group. Its relationship to the autogenerated template is not documented. Consider adding a description field clarifying when/where this type is used versus the autogenerated one.
- **Suggested Code:**
  ```yaml
  description: >
    Packed status information for exactly 4 fault response entries (1 byte).
    Used as the atomic packing unit within the autogenerated status record.
    See name_status_record.record.yaml for the full dynamically-sized record.
  ```
- **Severity:** **Low** — Documentation/maintainability concern.
