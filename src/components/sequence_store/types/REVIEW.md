# Code Review: `sequence_store/types`

**Reviewer:** Automated Ada/Adamant Review  
**Date:** 2026-03-01  
**Scope:** All `.ads`, `.record.yaml`, `.enums.yaml` files (excluding `build/`)

## Summary

This is a well-structured type package for a sequence store component. The record definitions are consistent between 32-bit and 64-bit variants, enumerations are thoughtfully designed (including the `Undefined` sentinel for MRAM corruption), and field descriptions are thorough. A few issues noted below.

---

## Findings

### 1. `Slot_Number` range exceeds meaningful use — **Low**

**File:** `sequence_store_types.ads`

```ada
type Slot_Number is new Natural range 0 .. 65_535;
```

`Slot_Number` is derived from `Natural` (32-bit on most targets) but constrained to `0 .. 65_535` and serialized as `U16` in all record YAMLs. The type itself is fine, but deriving from `Natural` means the base type is 32-bit, so stack variables consume 4 bytes while the wire format is 2 bytes. This is standard Adamant practice (the code generator handles packing), so this is informational only.

### 2. 32-bit / 64-bit record duplication — **Low**

**Files:** `32bit/*.record.yaml` vs `64bit/*.record.yaml`

All six record pairs are **byte-identical** in field definitions. The only difference is the target word-size path. If `Memory_Region.T` or `Sequence_Header.T` differ across targets this is justified, but the duplication creates a maintenance risk: a fix applied to one directory but not the other would silently diverge.

**Recommendation:** Add a comment in each YAML noting the twin, or consider a shared template if the Adamant build system supports it.

### 3. `Slot_Valid_Type.Undefined` at value 3 — design note — **Low**

**File:** `sequence_store_enums.enums.yaml`

The `Undefined` literal (value 3) exists to keep the 2-bit representation unconstrained, which is explicitly documented and correct for MRAM-backed storage. No issue — this is good defensive design. Just noting that any code consuming this enum **must** handle `Undefined` as an error/unexpected case; this should be verified at the component level.

### 4. `skip_validation` and `byte_image` on `Computed_Crc` — **Medium**

**Files:** `32bit/invalid_sequence_crc_info.record.yaml`, `64bit/invalid_sequence_crc_info.record.yaml`

```yaml
skip_validation: True
byte_image: True
```

`skip_validation: True` disables the autogenerated range/validity check for `Computed_Crc`. While CRC values intentionally span the full `U16` range (making validation meaningless), `skip_validation` also suppresses any future structural checks the code generator might add. This is acceptable but should be documented with a comment explaining *why* validation is skipped.

`byte_image: True` exposes raw byte access. Ensure this is intentional and that no endianness assumptions leak into consumers.

### 5. `Reserved` field in `sequence_store_slot_metadata` not zero-initialized — **Medium**

**File:** `sequence_store_slot_metadata.record.yaml`

```yaml
- name: Reserved
  description: Unused bits.
  type: Five_Bit_Pad_Type
  format: U5
```

The 5-bit reserved/padding field uses a custom `mod 2**5` type but no default value is specified. In MRAM-backed storage, uninitialized reserved bits could be non-zero, which is fine for reads, but when *constructing* records for telemetry or events, garbage in reserved bits can confuse ground tools or make binary diffs noisy.

**Recommendation:** Set a default of `0` if the Adamant record generator supports it, or document that the field is don't-care.

### 6. `sequence_store_memory_region_release` uses `Sequence_Store_Status` — **Low**

**File:** `32bit/sequence_store_memory_region_release.record.yaml` (and 64-bit twin)

The status field uses `Sequence_Store_Status` (which includes `Crc_Error`, `Length_Error`, etc.) for a *release* operation. Several of those literals are semantically meaningless for release. This isn't a defect — Ada enum representation means unused values simply won't appear — but a dedicated release-status enum would improve self-documentation.

### 7. No `.adb` files — **Info**

No Ada body files exist in this package. This is expected for a pure types/records package (all logic is in the generated code or parent component).

---

## Verdict

**No critical or high-severity issues found.** The package is clean, well-documented, and follows Adamant conventions. The medium findings (reserved-field initialization, skip_validation documentation) are worth addressing for long-term maintainability. The 32/64-bit duplication is the largest maintenance risk but is likely an Adamant framework constraint.
